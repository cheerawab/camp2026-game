# Backend Agent Guide

This guide applies to `server/`, the Go backend for the Camp 2026 game. It is written for coding agents and engineers who need to make backend changes quickly without losing the structure that keeps the project maintainable.

The backend is intentionally lightweight. Do not turn it into the full clean architecture style used by larger reference projects unless the project clearly grows into that need.

## Project Intent

This backend exists to support a short-lifecycle camp game. The main goal is to ship a clear, reliable API quickly.

Prioritize these values in this order:

1. Fast and understandable implementation.
2. Direct MongoDB access through explicit, request-scoped calls.
3. Clear HTTP structure where route groups and endpoint files are easy to find.
4. Enough tests to protect behavior.
5. Minimal abstractions until real complexity appears.

Avoid architecture that exists only for ceremony. A handler may call a MongoDB
collection directly for simple reads and writes. Add a service layer only when
it makes a use case easier to understand, test, or reuse.

## Current Backend Shape

The backend is a single Go module under `server/`.

```text
server/
  cmd/api/                  process entry point
  internal/app/             composition root and graceful lifecycle
  internal/config/          environment loading, defaults, validation
  internal/content/         TOML game content loader and in-memory store
  internal/http/            router, server, middleware, Swagger wiring
  internal/http/apimodel/   shared HTTP request and response DTOs
  internal/http/handler/    HTTP route groups
  internal/http/httpx/      JSON and problem response helpers
  internal/mongodb/         MongoDB client and readiness check
  docs/                     generated Swagger files and backend design docs
```

Important generated paths:

- `docs/docs.go`, `docs/swagger.json`, and `docs/swagger.yaml` are generated by `swag`.
- Never edit generated files by hand. Edit handler annotations, DTOs, or routes, then run `make generate`.
- Handwritten design docs, such as `docs/game-data-api-design.md`, may be edited normally.

## Commands

Run commands from `server/`.

```sh
make dev
```

Starts the API with Air hot reload using `.air.toml`.

```sh
make run
```

Starts the API once with `go run ./cmd/api`, without file watching.

```sh
make generate
```

Runs Swagger generation. Run this after changing handler Swagger annotations,
response DTOs, routes, or public API status codes.

```sh
make swagger
```

Regenerates only `docs/`.

```sh
make test
```

Runs `go test ./...`. This is the minimum verification before finishing backend code changes.

```sh
make tidy
```

Runs `go mod tidy`. Run this after adding, removing, or changing dependencies.

## Request Lifecycle

The current request path is intentionally direct:

```text
cmd/api/main.go
  -> internal/app.New
  -> config.Load
  -> mongodb.NewClient
  -> http.NewRouter
  -> handler group
  -> MongoDB collection call when needed
  -> httpx response
```

`cmd/api/main.go` owns process startup and signal handling. Keep it small. It should create the shutdown context, build the application, run it, and report startup/runtime failures.

`internal/app` is the composition root. It wires config, logger, MongoDB client,
router, and HTTP server. New cross-cutting dependencies should be created here
and passed into the router or handlers through explicit dependency structs.

`internal/http/router.go` owns middleware, API prefix routing, group registration, 404, 405, and Swagger route registration. It should not contain endpoint business logic.

## Routing Rules

The API prefix is:

```text
/api
```

Current public routes:

```text
GET /api/healthz
POST /api/auth/login
POST /api/auth/logout
GET /api/swagger.json
GET /api/docs
GET /api/docs/index.html
```

When adding a new route group, create a handler group folder and register it from `internal/http/router.go`. Keep route registration explicit. Do not hide routes behind package-level side effects.

## Handler Group Rules

Every route group gets one folder:

```text
internal/http/handler/<group>/
```

Every endpoint gets one file:

```text
internal/http/handler/<group>/<endpoint>.go
```

The group folder must contain a `handler.go` file. Keep `handler.go` focused on:

- `Dependencies`
- `Handler`
- `New`
- `RegisterRoutes`

Example:

```text
internal/http/handler/system/
  handler.go
  health.go
```

Do not put all endpoint logic into `handler.go`. Do not create a large `handlers.go` file for a whole group.

Contract stub endpoints are allowed while the API is being reviewed. A contract
stub must still be registered through the real router, include Swagger
annotations, and use the real request decoder and validator for JSON request
bodies. For action routes whose business logic is not implemented yet, return
`501 Not Implemented` through `httpx.WriteProblem`.

Endpoint files should contain only what that endpoint needs:

- Request DTOs, if any.
- Response DTOs, if endpoint-specific.
- Swagger annotations.
- The handler method.

If a response DTO is shared by several endpoints in the same group, it can live in a small shared file such as `response.go` inside that group. Do not create shared files prematurely.

## Adding A New Endpoint

Use this workflow:

1. Choose the route group name.
2. Create or reuse `internal/http/handler/<group>/`.
3. Add or update `handler.go` with constructor dependencies and `RegisterRoutes`.
4. Add one endpoint file named after the endpoint action.
5. Define request and response DTOs near the handler.
6. Use `httpx.WriteJSON` for success responses.
7. Use `httpx.WriteProblem` for error responses.
8. Add Swagger annotations above the handler method.
9. Register the group from `internal/http/router.go` if it is new.
10. Add or update router/handler tests.
11. Run `make generate`.
12. Run `make test`.

For simple database-backed endpoints, it is acceptable for the handler to call a
MongoDB collection directly with `r.Context()`.

Add a service package only when at least one of these is true:

- Multiple endpoints share the same use case logic.
- The logic has meaningful branching beyond request binding and response mapping.
- The operation needs a transaction or coordinates multiple queries.
- Testing the behavior through HTTP becomes awkward.
- Business language is clearer as a named use case method.

Do not add service, repository, domain, or interface layers by default.

## Handler Dependencies

Each handler group owns its own dependency struct. Keep dependencies explicit.

Good:

```go
type Dependencies struct {
    DB *mongo.Database
}
```

Avoid broad dependency bags that pass the whole app, config, database pool, or router into handlers.

If a handler needs a new dependency, add it to:

1. The group `Dependencies` struct.
2. The group `Handler` struct.
3. The group constructor.
4. The router registration wiring.
5. Tests that construct the router or handler.

## Responses And Errors

Use `internal/http/httpx` for HTTP response writing.

Success responses:

```go
httpx.WriteJSON(w, http.StatusOK, body)
```

Problem responses:

```go
httpx.WriteProblem(w, r, httpx.ServiceUnavailable("readiness check failed"))
```

`httpx.WriteProblem` writes `application/problem+json` with a conservative RFC 7807-style body.

Do not return raw database errors, stack traces, secrets, or internal package details in response bodies. Technical details belong in logs. Public errors should be short and stable.

Prefer these broad mappings:

- Invalid route: `404`
- Unsupported method: `405`
- Invalid request body or query: `400`
- Semantic validation error: `422`
- Contract exists but business logic is not built yet: `501`
- Conflict or duplicate resource: `409`
- Missing authentication, when auth exists later: `401`
- Authenticated but not allowed, when auth exists later: `403`
- Dependency unavailable: `503`
- Unexpected technical failure: `500`

Only add helpers to `httpx` when they are broadly useful. Do not create one-off helpers for a single endpoint.

## Request Validation

HTTP request validation uses `github.com/go-playground/validator/v10` through helpers in `internal/http/httpx`.

Use this standard handler flow for JSON requests:

```go
var body CreateThingRequest
if err := httpx.DecodeJSON(r, &body); err != nil {
    httpx.WriteProblem(w, r, err)
    return
}

if err := httpx.ValidateStruct(body); err != nil {
    httpx.WriteProblem(w, r, err)
    return
}
```

`DecodeJSON` handles transport/body parse errors and returns `400`:

- missing body
- malformed JSON
- unknown fields
- more than one JSON value

`ValidateStruct` handles semantic validation and returns `422` with `errors[]`.

Use validator tags on HTTP request DTOs:

```go
type CreatePlayersRequest struct {
    Players []PlayerInput `json:"players" validate:"required,min=1,max=20,dive"`
}

type PlayerInput struct {
    DisplayName string `json:"displayName" validate:"required,max=40"`
    TeamNumber  int    `json:"teamNumber" validate:"required,min=1,max=20"`
}
```

Use `dive` for slice/array element validation. Error locations should preserve array indexes:

```json
{
  "location": "body.players[0].displayName",
  "message": "displayName is required"
}
```

Keep custom normalization, such as trimming strings, close to the handler group in a small `validate.go` file. Normalize before calling `ValidateStruct` when whitespace should not count as input.

## Middleware

HTTP middleware is registered in `internal/http/router.go`.

Current middleware responsibilities:

- Request ID.
- Real client IP handling.
- Slash normalization.
- Request timeout.
- Panic recovery.
- Structured request logging.

Keep middleware generic. Feature-specific validation, database access, and use case behavior belong in handlers or future services, not middleware.

The recovery middleware must never expose panic values or stack traces to clients.

## Configuration

Configuration lives in `internal/config`.

Current environment keys:

```text
APP_ENV
SERVICE_NAME
APP_VERSION
LOG_LEVEL
HTTP_ADDR
REQUEST_TIMEOUT
HTTP_READ_HEADER_TIMEOUT
HTTP_READ_TIMEOUT
HTTP_WRITE_TIMEOUT
HTTP_IDLE_TIMEOUT
SHUTDOWN_TIMEOUT
CONTENT_DIR
MONGODB_URI
MONGODB_DATABASE
```

Defaults are defined in `internal/config/config.go`.

When adding a new env key:

1. Add it to `internal/config/config.go`.
2. Add a sensible default if local development should work without extra setup.
3. Validate it if invalid values would cause confusing runtime behavior.
4. Add it to `.env.example`.
5. Update this file if the key changes backend behavior or developer workflow.

Durations use Go duration strings such as `5s`, `10s`, `1m`, or `2h`.

The loader reads `.env` and `server/.env`. Do not commit real `.env` files.

## Database And Indexes

The backend uses MongoDB through the official Go driver.

The shared client and readiness check live in:

```text
internal/mongodb/
```

The application does not create collections, indexes, or seed data automatically
on startup. Add explicit setup scripts when those become necessary.

When adding database-backed behavior:

1. Define the collection and document shape close to the feature using it.
2. Pass `*mongo.Database` or a narrower dependency through the handler group.
3. Use `r.Context()` for all request-scoped database calls.
4. Map MongoDB documents to explicit HTTP DTOs.
5. Add tests around the behavior that consumes the collection.

Keep collection names and indexes documented near the feature or setup script
that owns them. Prefer small query result shapes and projections when handlers
do not need the full document.

## Swagger And API Docs

Swagger is docs-only. The project does not use generated server stubs.

Swagger source comes from comments above endpoint handlers. Generated files live in `docs/`. The human-facing API reference UI is a minimal Scalar HTML/JS page served by the backend; `swaggo/swag` owns the generated `swagger.json`.

Do not switch the project to `go-swagger` unless explicitly requested. The
current generation path is:

```text
Go handler annotations
  -> swag init
  -> docs/swagger.json
  -> GET /api/swagger.json
  -> Scalar at GET /api/docs
```

Use this command after changing endpoint annotations, routes, DTOs, or response status codes:

```sh
make swagger
```

or:

```sh
make generate
```

Swagger routes are served under the API prefix:

```text
GET /api/swagger.json
GET /api/docs
GET /api/docs/index.html
```

The docs HTML uses Scalar through jsDelivr:

```html
<script src="https://cdn.jsdelivr.net/npm/@scalar/api-reference"></script>
```

and mounts it with:

```js
Scalar.createApiReference('#app', {
  url: '/api/swagger.json',
})
```

Each endpoint should include at least:

- `@Summary`
- `@Description`
- `@Tags`
- `@Produce json`
- `@Success`
- `@Failure`, when applicable
- `@Router`

Keep Swagger DTO references accurate after moving types between packages.

## Logging

The backend uses the standard library `log/slog`.

The app logger is created in `internal/app`. HTTP request logging is middleware-level and records:

- Method.
- Path.
- Status.
- Duration in milliseconds.

Use structured key/value logging:

```go
log.Info("message", "key", value)
```

Do not log:

- Passwords.
- Tokens.
- Cookies.
- Database passwords.
- Secrets.
- Raw request bodies containing sensitive fields.
- Raw personal data unless the feature explicitly requires it and the team has accepted that risk.

Logs are for operators and debugging, not for returning technical details to clients.

## Security Rules

Current template has no auth yet. When auth is added later, keep the security boundary explicit.

Always follow these rules:

- Never commit real secrets.
- Never include real credentials in `.env.example`.
- Never expose raw infrastructure errors to clients.
- Never log secrets or auth material.
- Treat request body logging as unsafe by default.
- Prefer deny-by-default behavior for future protected routes.

For public endpoints, be conservative with error details. Do not reveal whether hidden or unauthorized resources exist once auth/resource ownership exists.

## Testing Rules

Tests use Go standard `testing`.

Minimum test expectations for a new endpoint:

- Success path returns the expected status.
- Response JSON has the expected important fields.
- Error path returns the expected status.
- Problem responses use `application/problem+json`.
- Router registration works through the real router when possible.

Router-level tests currently live in:

```text
internal/http/router_test.go
```

If a handler group grows enough to need focused tests, add tests beside the group:

```text
internal/http/handler/<group>/*_test.go
```

Do not test generated Swagger code directly. Test the behavior behind the API.

For DB-heavy behavior, prefer clear integration-style tests once collection and
index setup is established. Until then, keep tests focused on handler behavior
and error mapping.

Always run:

```sh
make test
```

before finishing backend code changes.

## Dependency Rules

Current direct runtime dependencies are intentionally small:

- `github.com/go-chi/chi/v5`
- `github.com/go-playground/validator/v10`
- `github.com/joho/godotenv`
- `github.com/swaggo/swag`
- `go.mongodb.org/mongo-driver/v2`

Do not add new dependencies unless standard library plus current dependencies are clearly not enough.

Before adding a dependency, consider:

- Can this be done clearly with the standard library?
- Is the package actively maintained?
- Does it introduce heavy transitive dependencies?
- Does it affect Docker image size or startup behavior?
- Is the dependency needed at runtime, or only for development?

After changing dependencies, run:

```sh
make tidy
make test
```

## Code Style

Use `gofmt` for all Go files.

Package names should be short, lowercase, and meaningful. Prefer names like:

- `config`
- `mongodb`
- `httpx`
- `system`

Avoid vague names like:

- `common`
- `utils`
- `helpers`
- `manager`

Use `context.Context` from incoming requests for request-scoped work:

```go
err := h.collection.FindOne(r.Context(), filter).Decode(&doc)
```

Do not use `context.Background()` inside request handlers for database calls.

Keep imports acyclic. Handler group packages live below `internal/http`, so shared response helpers are in `internal/http/httpx` to avoid importing the parent `internal/http` package.

## DTO Rules

Keep endpoint-specific HTTP DTOs close to the endpoint that uses them.

Good for endpoint-specific response:

```go
type ActivityVerificationResponse struct {
    VerificationID string `json:"verificationId"`
    PlayerID       string `json:"playerId"`
}
```

If multiple endpoints share a response shape inside the same group, move it to a group-local file such as `response.go`.

For cross-feature API contract DTOs used by Swagger and multiple handler
groups, place them under:

```text
internal/http/apimodel/
```

Keep `apimodel` limited to HTTP request/response shapes. It should not contain
database queries, business logic, service logic, or MongoDB document types.

Do not create global DTO packages unless several route groups truly share the same API type.

MongoDB document types are not automatically HTTP response types. It is
acceptable to return them only when the field names and JSON shape are
intentionally part of the public API. Otherwise, map to explicit response DTOs.

## When To Add A Service Layer

The default is no service layer.

Add a service package only when it solves a real problem. Good reasons:

- A use case spans multiple queries.
- A transaction is required.
- The same logic is used by multiple endpoints.
- Validation or state transitions are too large for a handler.
- Unit testing the use case separately from HTTP would materially improve clarity.

If a service is added, keep it small:

```text
internal/<feature>/
  service.go
  errors.go
  validate.go
```

or another simple structure that fits the feature. Do not introduce a full clean architecture folder tree automatically.

Handlers should still own HTTP binding and response mapping. Services should not import `net/http`, `chi`, or `httpx`.

## When To Add A Repository Layer

The default is no repository layer.

Add a repository only when direct MongoDB collection usage becomes noisy or
repeated. Good reasons:

- Repeated document-to-domain mapping.
- Repeated MongoDB error translation.
- Transaction composition across multiple collection calls.
- A feature needs a narrow persistence interface for tests.

Do not create one repository per table by default.

## Generated File Policy

Generated files are source artifacts, but they are not hand-authored.

Allowed:

- Commit generated files when they are part of the backend source tree.
- Regenerate them through `make generate`.
- Review diffs to confirm they match intended API changes.

Not allowed:

- Manual edits to generated Swagger files.
- Cosmetic churn in generated files unrelated to the task.

If generated output changes unexpectedly, investigate before committing.

## Docker

The backend Dockerfile builds `./cmd/api` into a small Alpine runtime image.

Keep Dockerfile behavior simple:

- Build from `server/go.mod` and `server/go.sum`.
- Copy `server/`.
- Build a single `/app/api` binary.
- Run as a non-root user.

Do not add development-only tools to the runtime image.

## Local Development

Use `.env.example` as the local template:

```sh
cp .env.example .env
```

Then adjust `CONTENT_DIR`, `MONGODB_URI`, or `MONGODB_DATABASE`.

The default `MONGODB_URI` is:

```text
mongodb://camp2026:camp2026@localhost:27017/camp2026?authSource=admin
```

The app requires a reachable database when starting because `internal/app.New`
opens and pings the MongoDB client. `/healthz` uses the same database readiness
check.

## Common Recipes

### Add a non-DB endpoint

1. Add `internal/http/handler/<group>/<endpoint>.go`.
2. Add the route in the group's `RegisterRoutes`.
3. Add Swagger annotations.
4. Use `httpx.DecodeJSON` and `httpx.ValidateStruct` for JSON request bodies.
5. Register the group in `router.go` if new.
6. Add router/handler tests.
7. Run `make generate`.
8. Run `make test`.

### Add a DB-backed endpoint

1. Define the MongoDB document shape and collection name near the feature.
2. Add handler endpoint file.
3. Call MongoDB with `r.Context()`.
4. Map result to an HTTP DTO.
5. Add Swagger annotations.
6. Add tests.
7. Run `make generate`.
8. Run `make test`.

### Add a new config key

1. Update `internal/config/config.go`.
2. Update `.env.example`.
3. Use the config from `internal/app` wiring.
4. Add tests if parsing or validation is non-trivial.
5. Update this file if the key changes the expected workflow.

### Add a new dependency

1. Confirm current stack cannot solve the problem cleanly.
2. Add the import/use.
3. Run `make tidy`.
4. Run `make test`.
5. Document the dependency if it changes backend conventions.

## What Not To Do

Do not:

- Recreate the large reference project's full clean architecture.
- Add `domain/application/infrastructure/transport` layers by default.
- Put many endpoints into one large handler file.
- Add one repository per table automatically.
- Add interfaces only for the sake of interfaces.
- Manually edit generated Swagger files.
- Return raw internal errors to clients.
- Log secrets, tokens, cookies, passwords, or raw sensitive request bodies.
- Add dependencies without a concrete need.
- Hide route registration in package `init` functions.
- Create MongoDB collections or indexes at app startup without an explicit design decision.
- Use `context.Background()` for request-scoped database work.

## Agent Workflow Checklist

Before editing:

1. Inspect the nearest existing package pattern.
2. Check whether the change affects handlers, MongoDB collections, Swagger, config, tests, or docs.
3. Keep the change scoped to the requested behavior.

While editing:

1. Follow the route group and endpoint file structure.
2. Keep DTOs near their endpoint.
3. Use `httpx` for all HTTP responses.
4. Use the official MongoDB driver consistently.
5. Avoid new abstractions unless they remove real complexity.

Before finishing:

1. Run `make generate` if Swagger annotations, DTOs, or routes changed.
2. Run `make test` if any Go code changed.
3. Review generated diffs.
4. Update `server/AGENTS.md` if architecture, commands, config, generation, or handler rules changed.
5. Mention any skipped verification in the final response.

This file should stay accurate. If future changes make any section wrong, update this guide in the same change.
